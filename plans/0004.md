## Goal

Make 1:1 text actually deliver between two running clients (desktop ↔ mobile), with honest connection status and recoverable sends.

## Changes

### Overlay ([network.rs](../src-tauri/src/network.rs))

- **mDNS** LAN discovery; `add_explicit_peer` on discover
- **Announce v2** (`vibe/announce/2`) includes dialable `listenAddrs` (LAN IPs + `10.0.2.2` for emulator → host); dial on receive
- **Direct room announce** on libp2p connect via `/vibe/room-announce/1` (breaks gossipsub chicken-and-egg)
- **identify** — dial addresses learned from connected peers
- **Small-network gossipsub** — `mesh_n(1)`, `flood_publish`, faster heartbeat
- **Subscribe** `vibe/msg` + `vibe/signal` per contact on add and `start_network`
- **Resilient publish** — log errors, do not kill swarm loop (`InsufficientPeers` is OK until a peer connects)
- **`overlay_peer_count`** command + `overlay-peers-changed` event

### WebRTC ([webrtc.ts](../src/lib/webrtc.ts))

- Google STUN: `stun.l.google.com:19302`, `stun1.l.google.com:19302`
- Gossipsub fallback if DC send fails; `resetTextTransport` on errors

### UI

- Banner when overlay peer count is 0
- **Connected** badge only when DC open and overlay peers > 0
- Toast + restore draft on send failure

## Manual test

1. Start Vibe on desktop and mobile (or emulator) on the same LAN
2. **Join the same room** on both; add each other from the room list
3. Open chat; confirm overlay banner clears (peer count > 0)
4. Exchange several messages both ways
5. Leave room on one side → send shows error toast

## Known limits

- Contact-only (no room) still has no discovery path
- STUN is hardcoded (M0); SPEC §9.4 user-toggle deferred
- No Kademlia bootstrap yet
- iOS: `libp2p-mdns` requires `SystemConfiguration.framework` in `bundle.iOS.frameworks` ([tauri.conf.json](../src-tauri/tauri.conf.json))
